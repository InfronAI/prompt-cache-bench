Infron 与 OpenRouter 路由策略下的提示词缓存、成本与性能 A/B 基准研究

摘要与结论大纲

关键词：提示词缓存；A/B 测试；服务商路由；缓存亲和性；端到端E2E时延；流式TTFT；吞吐量；成本归因；DeepSeek V4 Flash

摘要

本报告以 deepseek/deepseek-v4-flash 为对象，对比 Infron 与 OpenRouter 在 提示词缓存场景下的路由策略、缓存命中、实际成本、吞吐量、流式TTFT和端到端E2E时延。实验包含 4 个实验组、每组 50 轮，覆盖 3 种 路由排序策略。经过异常 usage、HTTP 异常和 A/B 输入 tokens 不一致样本剔除后，最终保留 364 个严格配对样本、1456 次请求级观测记录，剔除 472 条记录。

核心结论是：在 usage.prompt_tokens 完全一致的样本中，Infron 在所有路由模式下，Token 级缓存命中率都更高；Infron 在 price 和 latency 模式下，实际成本更低，OpenRouter 在 throughput 模式下，实际成本更低；OpenRouter 在所有路由模式下，吞吐量都更高；Infron 在 latency 模式下，端到端E2E时延更低，OpenRouter 在 throughput 和 price 模式下，端到端E2E时延更低；Infron 在 latency 模式下，流式TTFT更低，OpenRouter 在 throughput 和 price 模式下，流式TTFT更低。整体看，Infron 的优势集中在缓存复用、成本控制和 时延优先 下的低端到端E2E时延路径，OpenRouter 的优势集中在吞吐、流式TTFT 和部分模式的端到端E2E时延表现。平台选择应围绕业务目标展开，单一指标不足以代表整体效果。

1. 引言：背景、研究问题与贡献

本实验评估同一 OpenAI 兼容 Chat Completions 请求在 Infron 与 OpenRouter 两个平台上的 提示词缓存 表现。评估重点是：在输入条件严格一致时，不同服务商路由排序策略会如何影响缓存命中、实际成本、吞吐量和端到端E2E时延。

提示词缓存 对生产业务的核心价值在于：当业务请求包含稳定系统提示词、长上下文模板、RAG 前缀、工具说明或固定工作流指令时，第二次及后续请求理论上可以复用已处理的输入 token，从而降低单位请求成本，并可能改善整体服务稳定性。本实验通过“两次相同 提示词请求”的方式构造可重复观测场景，用第二次请求的 缓存读取 tokens 衡量缓存收益。

本报告回答三个问题：第一，在相同 请求载荷 和相同 usage.prompt_tokens 口径下，Infron 与 OpenRouter 的缓存命中和成本表现有何差异；第二，不同 路由排序（throughput、price、latency）下速度、成本和缓存如何变化；第三，从可观测 遥测数据 看，两个平台的路由选择如何影响最终结果。

2. 方法：实验设计、数据集构造与控制变量

2.1 数据集生成方法

实验数据集由脚本自动生成，共覆盖 3 种 路由排序、2 个平台、4 个实验组、每组 50 轮。每一轮包含两次完全相同的 chat/completions 请求：第一次用于建立或触发缓存写入，第二次用于观测缓存读取。每个 路由排序都使用固定 请求载荷，并记录该 请求载荷 的 SHA256，以便验证请求内容没有漂移。

提示词使用脚本内置的代表性业务模板，覆盖 RAG 客服、Agent 工具说明、营销自动化和代码审查四类稳定长上下文场景。每一轮在同一 group/round 下向 Infron 与 OpenRouter 发送完全相同的 消息，用于观察真实路由、缓存、成本、吞吐和端到端E2E时延差异。

2.2 控制变量方法

A/B 测试的基本配对单元是同一 sort/group/round 下的 Infron 记录和 OpenRouter 记录。只有当两边 第一次请求 与 第二次请求 的 usage.prompt_tokens 完全一致时，该配对才进入最终统计；任何 HTTP 非 200、请求异常、usage.prompt_tokens <= 0 或 A/B 输入 token 不一致的记录都会被剔除。这保证了成本、缓存命中率、吞吐量和端到端E2E时延的对比建立在同等输入规模上。

本报告中的总 输入 Tokens 严格取自响应返回的 usage.prompt_tokens，不使用本地 分词器 估算值。原因是 服务商的真实处理、缓存和计费口径最终以响应 usage 为准。通过使用响应 usage 并执行 A/B 配对一致性过滤，实验避免了 分词器 差异、服务端 提示词包装和异常 usage 上报带来的偏差。

6. Infron 技术架构与缓存/成本机制解释

本节使用本次 benchmark 的可观测结果解释 Infron 在高 缓存命中率 与成本控制上的工程路径。需要说明的是，报告没有采集 Infron 内部私有 路由轨迹；因此下文把响应中真实返回的服务商分布、缓存读取 tokens、成本明细 和 latency/吞吐量 指标作为证据，用架构图解释这些结果背后的合理机制。

6.1 多 服务商 路由与可观测控制面

Infron 对外提供 OpenAI 兼容 API，对内需要在多个上游服务商、模型部署和路由策略之间做选择。对 提示词缓存 工作负载而言，路由层不只是选择一个可用 服务商，还需要同时考虑缓存亲和性、健康状态、成本、吞吐和端到端E2E时延目标。

图 12：Infron 多 服务商 路由与缓存控制面。该图用于说明请求从统一 API 入口进入后，路由控制面如何在健康状态、策略目标、服务商选择和缓存域之间形成决策链路。

本次实验中，Infron 在不同 路由排序下呈现出清晰的服务商分布：throughput 主要路由到 alibaba/cn，price 主要路由到 gmicloud，latency 主要路由到 novita。这种模式说明路由结果不是完全随机扩散，而是围绕路由目标形成了较稳定的服务商选择。稳定的服务商选择是高缓存命中率的前提，因为 提示词 cache 通常与具体 服务商、模型部署或缓存域绑定。

6.2 服务商 Stick 与 缓存亲和性

服务商粘滞 是多 服务商 网关中的缓存亲和策略：当请求具有相同或高度稳定的 提示词 prefix 时，路由层倾向于把同一类请求送往同一个健康服务商 或缓存域，以减少缓存碎片化。它不等于固定永不切换 服务商；当上游不可用、限流或 SLA 风险升高时，路由仍应回退到其他健康路径。

图 13：服务商粘滞与缓存亲和性 机制。该图表达的是工程机制假设：同类请求在健康服务商集合内保持缓存亲和，可减少跨 服务商缓存域 的缓存碎片。

这解释了 Infron 在本次实验中的高 Token 级命中率：在 throughput 与 latency 两个模式下，Infron 的第二次请求 Token 级缓存命中率约为 94.42%，OpenRouter 约为 44%-45%。对于相同 stable prefix 的连续双请求，若路由落在同一缓存域，第二次请求更容易读取第一次请求写入或刷新后的 KV/cache 状态；若请求在多个 服务商 或部署之间分散，同样的 提示词 也可能需要分别暖缓存，从而降低整体 缓存读取 tokens。

6.3 成本控制路径

成本控制来自两层叠加：第一层是缓存命中降低重复预填充 的有效处理成本；第二层是 服务商路由 在健康服务商集合内选择更合适的成本路径。本次实验中，Infron 在三个路由模式下的实际成本均低于 OpenRouter，同时 Token 级缓存命中率显著更高，说明缓存亲和和 服务商选择共同影响了单位请求成本。

9. 讨论：业务价值、适用边界与工程启示

三种 路由排序对应不同业务目标，需要结合缓存、成本、吞吐和端到端E2E时延一起判断。throughput 更适合批处理、异步生成、长文本生产和离线任务；price 更适合高频低毛利调用、固定模板请求、客服/营销自动化等成本敏感场景；latency 更适合交互式产品、Agent 工具调用链、实时辅助写作和用户等待成本较高的场景。

路由模式主要业务目标本轮数据体现适用场景注意事项
throughput最大化单位时间输出能力缓存 Infron 占优，成本 OpenRouter 占优，吞吐量 OpenRouter 占优，端到端E2E时延 OpenRouter 占优批量内容生成、离线摘要、后台数据加工速度和成本同时较强，但仍需确认缓存命中稳定性
price最小化单位请求和单位 token 成本缓存 Infron 占优，成本 Infron 占优，吞吐量 OpenRouter 占优，端到端E2E时延 OpenRouter 占优高频模板化请求、客服自动化、营销触达、RAG 固定前缀适合成本敏感任务，但需检查吞吐是否满足 SLA
latency最小化用户可感知等待时间缓存 Infron 占优，成本 Infron 占优，吞吐量 OpenRouter 占优，端到端E2E时延 Infron 占优在线聊天、Agent 调用链、IDE/写作辅助、实时运营工具更适合成本和体验受控的在线业务，但吞吐可能不是最优

从业务决策角度看，提示词缓存 的价值不只体现在单次请求省钱，而是体现在大规模重复上下文请求的边际成本下降。若业务请求结构高度模板化，应优先关注 Token 级命中率和实际成本；若业务以用户实时体验为核心，应同时约束 latency；若业务为后台批量生成，则 吞吐量 可能比单请求端到端E2E时延 更重要。

因此，本实验的推荐读法是：先确认 输入 Tokens 是否完全可比，再按业务目标选择主指标，最后检查其他指标是否出现不可接受的副作用。例如某个平台吞吐更高但缓存命中显著较低，可能适合批处理，却未必适合需要稳定成本结构的高频在线业务。
